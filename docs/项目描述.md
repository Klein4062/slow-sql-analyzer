# 执行计划自动分析工具

> **slow-sql-analyzer** —— PostgreSQL / openGauss 查询计划优化分析器

## 摘要

执行计划自动分析工具（`slow-sql-analyzer`）是一个面向 DBA、后端工程师与 SRE 的**数据库查询性能诊断工具**。它接收一条 **SQL 查询**或一份 **EXPLAIN 执行计划**，自动判断当前计划是否最优、定位性能反模式，并给出**可直接执行的优化建议**（缺失索引的 `CREATE INDEX`、`work_mem` 调优值、`ANALYZE` 刷新统计等）。

工具提供 **CLI、HTTP API、可视化网页**三种使用方式，支持 **离线解析**（粘贴计划）与 **实时连库**（执行 EXPLAIN）两种数据来源，覆盖 **PostgreSQL 与 openGauss**（含列存/向量化引擎），自动识别 **JSON 与文本**两种 EXPLAIN 格式。分析层为纯函数、零 IO，可完整单元测试；发布为**单个完全静态的二进制**，可零依赖部署到内网。已用真实 PostgreSQL 17 闭环验证（无索引查询建议建索引后，14.97ms → 1.23ms，约 12× 提升）。

## 创意详情

### 背景描述

数据库慢查询是线上性能问题最主要的来源之一，而 **`EXPLAIN` 执行计划**是诊断慢查询的核心入口。然而在真实工程中，从一份执行计划到「该改什么」之间存在显著的经验鸿沟：

- **计划晦涩难读**：执行计划是一棵含数十种节点类型、估算/实际行数、代价、缓冲、循环次数等指标的多层树，非资深 DBA 很难快速读懂，更难判断「当前计划是否最优」。
- **问题定位依赖专家经验**：哪些反模式（全表扫描、基数误估、磁盘排序、哈希溢出、嵌套循环重扫、低效过滤……）真正拖慢了查询、根因是缺索引、统计过时还是参数配置，需要深厚的优化经验。
- **修复建议需要手工推导**：即便看懂了问题，给出可落地的 `CREATE INDEX`、`work_mem` 取值、`ANALYZE` 仍然需要人工从计划字段里提炼，耗时且易错。
- **内网/受限环境难以下手**：许多生产库处于内网，无法随意装客户端、连公网拉依赖，或只能通过堡垒机/自带 `psql`、`gsql` 访问；统计信息是否过时也无从直观判断。
- **多数据库、多格式**：openGauss 与 PostgreSQL 计划结构相近但存在列存/向量化节点差异；EXPLAIN 既有 JSON 也有文本格式，工具往往只支持其一。

### 解决的问题

本工具系统性地解决上述痛点：

1. **「计划是否最优 → 问题在哪 → 怎么改」三连问自动化**：把读计划、判反模式、给建议的专家流程固化为可重复执行的规则引擎。
2. **覆盖最常见的 9 类性能反模式**（见方案描述），按严重度（critical/warning/info）分级，避免噪声。
3. **自动产出可执行建议**：从过滤/索引条件启发式提取列、合成 `CREATE INDEX`；按最大溢出量估算 `work_mem`；对统计过时或误估的表给出 `ANALYZE`；均可在 CLI/网页上一键复制。
4. **适配内网与受限环境**：内置 pgx 驱动（编译进二进制，目标机零客户端安装）+ `command` 连接器（用 `psql`/`gsql`/堡垒机脚本跑 EXPLAIN）+ 离线粘贴三种取数方式；单静态二进制拷贝即跑。
5. **主动检测统计信息新鲜度**：实时模式查询 `pg_stat_user_tables`，结合「修改占比」与「估算 vs 实际偏差」双信号，提前提示「该 ANALYZE 了」（病因检测 + 交叉印证 + 离线推断三级置信）。
6. **兼容 openGauss 列存/向量化**：识别 `CStore Scan`、`Vec Seq Scan`、`Vec Hash`、`VecAgg`、`Vec Nestloop` 等节点，9 条规则对列存计划同样生效。
7. **双格式兼容**：自动识别 EXPLAIN 的 JSON 与文本格式，降低使用门槛。

## 方案描述

### 总体架构

严格分层，数据单向流动，**分析层是纯函数、零 IO、可完整单元测试**：

```
source（取计划） → plan（解析 + 计划树） → analyzer（纯规则引擎） → report（渲染）
```

- **`internal/source`**：`PlanSource.Fetch()` 三实现——`PostgresSource`（内置 pgx 驱动）、`CommandSource`（shell 调 `psql`/`gsql`）、`FileSource`（离线文件/stdin）。**只有 source 做 IO**；实时统计新鲜度在此查得并附到 `PlanResult.TableStats`，再由纯规则消费。
- **`internal/plan`**：`PlanNode` 映射 EXPLAIN（JSON 自定义反序列化 + 文本启发式解析），节点判定 helper 同时覆盖 PG 与 openGauss 的行存/列存/向量化形态。
- **`internal/analyzer`**：`Rule` 接口 + 编排器，按严重度排序 findings。
- **`internal/rules`**：每条规则一个文件，是 `*AnalysisContext` 的纯函数。
- **`internal/advise`**：findings → 可执行 SQL 动作。
- **`internal/report`**：`text`（彩色）与 `json` 共用一个 `Model`，JSON 含序列化计划树供网页标注。
- **`internal/api`**（chi）+ **`internal/cli`**（cobra）：薄界面层；网页为单个 `go:embed` 内嵌 HTML。

### 规则引擎（9 条检测规则）

| 规则 | 触发 | 建议 |
|---|---|---|
| `SeqScanLargeTable` | 大表全表/全列扫描 | 过滤列建索引 |
| `CardinalityMisestimate` | 估算 vs 实际行数偏差过大 | `ANALYZE` / `CREATE STATISTICS` |
| `DiskSort` | 排序溢出磁盘 | 调大 `work_mem` / 建排序索引 |
| `HashSpill` | Hash 批次 > 1（溢出） | 调大 `work_mem` |
| `NestedLoopExpensiveInner` | Nested Loop 重扫昂贵内表 | 内表建索引 / 换 join |
| `InefficientFilter` | 过滤丢弃绝大多数行却未走索引 | 过滤列建索引 |
| `LowBufferHitRatio` | 缓冲命中率低 | 调大 `shared_buffers` |
| `Hotspot` | 节点独占执行时间过高 | 指出瓶颈 |
| `StaleStatistics` | 统计过时/不足（实时查系统表 / 离线推断） | `ANALYZE` 刷新统计 |

依赖实际统计的规则在「仅估算」计划下自动跳过并提示。

### 数据来源与连接器

- **实时 pgx**：内置驱动直连，默认在 `BEGIN READ ONLY` 事务内执行 `EXPLAIN (ANALYZE, BUFFERS)`，写语句被服务器拒绝。
- **实时 command**：`--exec` 命令模板（`{dsn}`/`{sql}` 占位符或 `$SSA_*` 环境变量），驱动 `psql`/`gsql`/堡垒机脚本，适配受限内网。
- **离线**：粘贴 JSON 或文本格式 EXPLAIN，无需数据库。

### 可执行建议合成

`advise` 把 findings 转成可直接复制执行的 SQL：按表聚合去重的 `CREATE INDEX`（启发式列提取，非完整 SQL 解析，明确标注需复核）、按最大溢出量 1.5× 估算的 `SET work_mem`、针对过时/误估表的 `ANALYZE`。

### 交互界面

- **CLI**：`plan`（离线）、`analyze`（实时，`--connector pgx|command`、`--exec`）、`serve`（HTTP+网页）、`version`；全局 flag 含 `--format text|json`、`--disable-rule`、各规则阈值。
- **HTTP API**：`POST /v1/plan`、`POST /v1/analyze`、`GET /v1/rules`、`GET /rules`、`GET /healthz`。
- **可视化网页**（`GET /`）：实时/离线双模式、带严重度标注且可点击跳转的计划树、分色 findings、一键复制建议动作；另有 `/rules` 规则说明页（通用/实时独有/离线独有三类）。

### 安全设计

实时模式默认只读事务 + `statement_timeout`；写语句默认拒绝，`--allow-writes` 才放开且仍在 `ROLLBACK` 事务内（永不提交）；`--no-analyze` 用于不允许执行的生产库。HTTP 服务建议配只读 DB 账号。

### 部署与分发

`CGO_ENABLED=0` 交叉编译为**单个完全静态二进制**（linux/darwin/windows × amd64/arm64），目标机零依赖（不需 Go/Python/`libpq`/`psql`）。air-gapped 构建用 `make vendor`；通过 GitHub Release 发布二进制 + SHA256SUMS。

### 验证

- 真实 PostgreSQL 17 闭环：无索引倾斜列查询 → 检出并建议 `idx (status)`，建索引后 14.97ms → 1.23ms（约 12×），写语句守卫与回滚安全已验证。
- openGauss：行存计划直接兼容；列存（Vec/CStore）节点识别后规则同样触发。
- 测试：单测覆盖各包（config/analyzer/advise 达 90%+），集成测试（`-tags=integration`）覆盖 live 路径，GitHub Actions CI 硬门禁。

### 技术栈

Go 1.26；`pgx/v5`（驱动）、`cobra`（CLI）、`chi`（HTTP）、`fatih/color`（终端彩色）、`go:embed`（网页）。标准库 `encoding/json` 处理 EXPLAIN。
